Step 01

ROI 기준선 구성

첫 단계는 배경이 많이 포함된 원본 이미지를 그대로 사용하지 않고, 검사 대상 영역을 안정적으로 잘라내는 기준선을 만드는 것이었습니다. 배경색 추정, AB/LAB distance map 계산, foreground mask 후보 생성, 최종 ROI 선택 과정을 통해 ROI-only baseline을 먼저 구성했습니다.

  • 이미지 테두리에서 배경색 추정
  • AB distance / LAB distance map 계산
  • threshold + mask cleanup으로 foreground 후보 생성
  • 최종 foreground mask와 bounding box를 선택해 crop
pcb4 ROI 예시
cashew ROI 예시

이 단계에서 강조한 포인트는 모델을 돌리기 전에 입력을 먼저 정리했다는 점입니다. 실제 검사에서는 배경이 많이 섞이면 score 분포와 heatmap 해석이 흔들릴 수 있기 때문에, ROI 기준선을 먼저 만든 흐름 자체가 운영 관점의 설계라고 볼 수 있습니다.

Step 02

CLAHE 비교와 정량·정성 분석

ROI-only 기준선을 만든 뒤에는, 같은 ROI에서 CLAHE를 추가했을 때 anomaly separation이 실제로 나아지는지 비교했습니다. 이 단계에서는 단순히 대비를 올리는 것이 아니라 score 분포, threshold, FP/FN 변화, representative heatmap/overlay를 함께 봤습니다.

Category Version AUROC Threshold Precision Recall F1
pcb4 ROI-only 0.9700 0.0756 0.7975 0.63 0.7039
pcb4 ROI+CLAHE 0.9748 0.0808 0.7931 0.69 0.7380
cashew ROI-only 0.9695 0.0554 0.7611 0.86 0.8075
cashew ROI+CLAHE 0.9533 0.0577 0.7217 0.83 0.7721

비교 결과는 카테고리별로 달랐습니다. pcb4는 CLAHE가 약한 결함 대비를 강화해 Recall과 F1을 개선했고, cashew는 ROI-only가 더 안정적이었습니다. 즉, 전처리는 무조건 강하게 넣는 것이 아니라 대상의 texture 특성에 따라 선택해야 한다는 점을 보여주도록 구성했습니다.

Step 03

추가 필터 실험과 최종 버전 선택

CLAHE 비교 이후에도 운영 관점에서는 false positive가 충분히 낮지 않았기 때문에, 카테고리별로 가벼운 필터를 추가한 extension experiments를 진행했습니다. 전체 파이프라인을 다시 바꾸기보다, 선택된 전처리 위에 median·gaussian·bilateral을 붙여 실제 운영 균형을 다시 비교했습니다.

pcb4 실험군

  • CLAHE + MedianBlur(3x3)
  • CLAHE + GaussianBlur(3x3)
  • CLAHE + MedianBlur(5x5)
  • CLAHE + BilateralFilter

cashew 실험군

  • ROI-only + MedianBlur(3x3)
  • ROI-only + BilateralFilter
  • ROI-only + GaussianBlur(3x3)
  • ROI-only + MedianBlur(5x5)
Category Selected Candidate 선정 이유
pcb4 CLAHE + Median3 F1 / FP / FN 균형이 가장 안정적
cashew ROI-only + Median5 high-recall 운영에서 더 유리한 후보

이 단계는 단순히 최고 점수 모델을 고르는 과정이 아니라, 실제 운영 정책에 맞는 후보를 선택하는 단계로 보이게 만들었습니다. 그래서 포트폴리오에서도 "왜 이 버전을 최종 버전으로 남겼는가"가 보이도록 표와 설명을 함께 넣었습니다.

Step 04

운영 임계값 재설정

비교 단계에서는 F1-optimal threshold를 사용했지만, 실제 검사 운영에서는 defect leakage를 더 중요하게 보기 때문에 final operation 단계에서는 recall-priority threshold로 다시 조정했습니다.

Category Version Threshold Type Threshold Precision Recall FP FN
pcb4 CLAHE + Median3 F1-optimal 0.0790273696 0.7500 0.7500 25 25
pcb4 CLAHE + Median3 Recall-priority 0.0750242769 0.5882 0.9000 63 10
cashew ROI-only + Median5 F1-optimal 0.0516819507 0.7258 0.9000 34 10
cashew ROI-only + Median5 Recall-priority 0.0498208888 0.6573 0.9400 49 6

여기서 보여주고 싶은 핵심은 비교용 기준과 운영용 기준을 분리했다는 점입니다. 공정 운영에서는 false positive가 늘더라도 false negative를 더 줄이는 판단이 필요할 수 있기 때문에, threshold를 두 단계로 다뤘다는 점이 이 프로젝트의 실무적인 포인트입니다.

Step 05

운영 FP/FN 사례 시각화

최종 operating threshold가 실제로 어떤 식으로 동작하는지 보여주기 위해 representative false positive / false negative 사례를 따로 정리했습니다. 숫자만 나열하는 대신, input·overlay·heatmap을 함께 보여줘서 판정 흐름을 설명할 수 있게 구성했습니다.

Step 06

Drift Monitoring · FastAPI · MFC Demo 연결

최종 버전과 threshold를 정한 뒤에는 새 배치가 들어왔을 때의 drift monitoring 기준, raw image를 받아 결과를 반환하는 FastAPI endpoint, 그리고 MFC 클라이언트 연동까지 정리했습니다. 그래서 이 프로젝트는 모델 비교에서 끝나는 프로젝트가 아니라, 실제 사용 흐름을 보여주는 포트폴리오로 확장됩니다.

Drift Monitoring

score distribution, gray/gradient statistics, predicted anomaly ratio를 기준으로 weak / moderate / severe drift를 나누고 retrain trigger skeleton까지 연결했습니다.

FastAPI Serving

raw image 업로드 → ROI → category별 전처리 → anomaly score / overlay / heatmap → threshold decision → JSON 응답 흐름으로 구성했습니다.
